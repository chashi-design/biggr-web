# AGENTS.md – Landing Page for Biggr App

## Mission / Goal
- このリポジトリは、**トレーニングログを記録・閲覧できる iOS アプリ「Biggr」**のランディングページ（LP）を開発する。
- 目的：ユーザーが **Biggr の価値を短時間で理解**し、**App Store に遷移して入手**したくなる状態をつくる。
- 成功指標（KPIの例）：
  - Primary CTA（App Storeで入手）のクリック数 / クリック率
  - App Store 遷移数

---

## Target Audience / ペルソナ
- 想定ユーザー：日々の筋トレ・ワークアウトを **手軽に記録**したい人
- 主な課題：
  - 手帳だと続きにくい。スマホで完結したい
  - 既存アプリは操作が重い／機能が多すぎて使い続けられない
- 主な利用シーン：筋トレ・ワークアウト中（**数秒で入力したい**）
- 対象デバイス / 地域：iOS / 全世界

---

## Value Proposition / 主要メッセージ
- 1行ベネフィット：**迷わず、すぐに記録できる**
- 差別化ポイント：入力・振り返りの動線を最短にした、シンプルで美しいアプリ体験
- 競合比較の軸：使いやすさ / 操作の軽さ / 迷いにくさ / 美しさ

---

## Key Features / 主要機能（訴求優先順）
1) **カレンダーからすぐ記録できる**  
2) **履歴をすぐ振り返れる**  
3) **iCloud に自動保存される**（※実態に合わせて表現する。誇張しない）  
4) **種目をすぐ検索できる**（例：YouTube検索）  
5) **お気に入りで入力を短縮できる**

- 機能の根拠（スクショ / 事例）：
  - 各機能に対応するスクリーンショットを最低1枚ずつ用意する
  - 「便利そう」ではなく「画面で理解できる」構成にする

---

## CTA設計
- Primary CTA（最優先導線）：
  - 文言：**「App Storeで入手」**（全ページで統一）
  - 目的：App Store への遷移
- Secondary CTA（補助導線）：
  - 文言例：**「特徴を見る」**（ページ内アンカー）
- 主要CTAの配置方針：
  - Hero（ファーストビュー）
  - ベネフィット/機能の後（中盤）
  - ページ下部（最終CTA）

---

## トーン & ボイス指針（Appleのように）
UI文言、オンボーディング、ヘルプ、エラーメッセージ、設定画面、告知文など、すべての文章は以下に従うこと。

### 1. 分かりやすく、人間らしく
- ふだんの言葉で書く。
- 1文は短くする。
- 専門用語はできるだけ避ける。
- 必要な場合のみ最小限に使い、直感的な言い回しにする。

### 2. 落ち着いていて、押しつけない
- 静かな自信を持った文にする（過剰に盛り上げない）。
- 感情を煽る表現や、売り込みっぽい表現は避ける。
- ！や絵文字は使わない。
- 不必要に謝りすぎない、長く説明しすぎない。

### 3. うまさより、明確さ
- 言葉遊び・冗談・流行語は使わない。
- 「気の利いた表現」より「誤解されない表現」を優先する。
- もっと短くできるなら短くする。

### 4. 行動が伝わる、直接的な言い方
- 何が起きるかが一目で分かる動詞を使う。
- 「〜してみましょう」「〜すると良いかも」は避ける。
- 例：「トレーニングを開始」＞「トレーニングを始めましょう」

### 5. 中立で、丁寧
- ユーザーを子ども扱いしない。
- ユーザーの感情や意図を決めつけない（努力・やる気等の前提を置かない）。
- 誘導的な励まし表現は避ける。

### 6. 用語の一貫性
- 同じ概念には同じ言葉を使う。
- “表現のバリエーション”のために言い換えない。
- 一般的に通じる語を優先し、造語や独自呼称は避ける。

### 7. 句読点・記号は最小限
- 「！」は使わない。
- 読点（、）を増やしすぎない。
- 短いラベルは句点（。）不要でよい。

### 8. エラー／システム文言
- 何が起きたかを短く伝える。
- 可能なら次にできる操作を添える。
- ユーザーのせいにしない。

例：
- 「保存できませんでした。もう一度試してください。」
- 「インターネットに接続できません。」

## Landing Page Content Requirements（App紹介LPの必須セクション）

### 目的
- 初見で「何のアプリか」「誰のためか」「なぜ良いか」を数秒で理解できる構成にする。
- 文章は短く、機能より“得られる結果（ベネフィット）”を先に伝える。
- Appleのように落ち着いたトーンで、過剰な主張や煽りを避ける。

### 必須セクション（推奨順）

#### 1) Header
- アプリアイコン
- アプリ名（表記ゆれ禁止）
- ダウンロードリンク（App Store）

#### 2) Hero（ファーストビュー）
- キャッチフレーズ（価値の核を1行）
- サブコピー（具体化：1〜2行）
- 主要CTA（「App Storeで入手」）※Headerと同じ文言で統一
- 実画面の画像（最重要：実際のUIを見せる）

#### 3) 課題提示（ユーザーの困りごと）
- 主観（「今までになかった」等）ではなく、ユーザー体験の困りごとで書く（2〜4個）
  - 例：記録が面倒で続かない / 見返しづらい / 種目のやり方を毎回調べるのが手間

#### 4) ベネフィット（3つに絞る）
- まず結果を短く提示する（見出し + 1文）
- 各ベネフィットに対応するスクリーンショットを配置する
  - 例：カレンダーで迷わず記録 / 振り返りがすぐできる / 種目の検索・登録がスムーズ

#### 5) 使い方（3ステップ）
- 導入障壁を下げるため、手順を最小で示す
  - 例：記録する → 振り返る → 続ける

#### 6) 機能詳細（根拠として列挙）
- 上のベネフィットを支える“具体機能”として整理して書く
  - 例：
    - カレンダーからサクッと記録
    - 日々の記録を簡単に振り返る
    - 種目をすぐ検索できる（YouTube検索等）
    - iCloudストレージに自動保存
    - お気に入り種目で素早く記録

#### 7) 安心・信頼（Trust）
- 誤解されやすい点／不安を先に潰す（短く）
  - データ保存（例：iCloudに自動保存）
  - プライバシー方針（要約を1〜2行）
  - お問い合わせ導線

#### 8) FAQ（CTA直前）
- 迷いを解消するため、最終CTAの直前に置く（5〜8項目目安）
  - 例：データ保存先 / 機種変更 / オフライン / 広告 / 対応OS

#### 9) 最終CTA
- Heroと同じCTA文言・同じ導線で統一
- 押しつけない短い一文まで（任意）

#### 10) Footer（リンク集）
- ダウンロードリンク（App Store）
- プライバシーポリシー（別ページ）
- 利用規約（別ページ）
- FAQ（別ページ or ページ内アンカー）
- お問い合わせ
- コピーライト
- 言語切り替え（日本語 / 英語）

### LPコピーの書き方ルール（Apple寄り）
- 「機能」より「得られる結果」を先に書く（ベネフィット → 根拠の順）
- 1ブロック＝1メッセージ。欲張って詰め込まない
- 1文は短く、句読点は最小限
- 感嘆符、絵文字、煽り、過剰な断定（世界一/唯一/史上初等）は避ける
- 主語がアプリ側の主張ではなく、ユーザー体験に寄せる
- 比較最上級（「最も〜」等）は原則使わない（使う場合は根拠が必要）

## SEO / Meta（LP）- Biggr

### タイトル（title）
- 目的：検索結果で「何のページか」を即理解させる
- ルール：
  - 先頭に主要価値（筋トレ/ワークアウト記録）を置く
  - 末尾にブランド名（Biggr）
  - 30〜40文字程度を目安
  - 誇張表現（世界一/完全/唯一等）は避ける

推奨フォーマット：
- 「筋トレ記録をシンプルに。カレンダーで続く｜Biggr」
- 「ワークアウト記録アプリ｜カレンダーで簡単に振り返り｜Biggr」

### ディスクリプション（meta description）
- 目的：クリック率を上げる要約（ページ内容の約束）
- ルール：
  - 80〜120文字程度を目安
  - ベネフィット → 主要機能（根拠）→ 信頼（iCloud等）の順
  - 落ち着いたトーンで、過剰な断定や煽りを避ける

推奨フォーマット：
- 「{誰向け}のための{何のアプリ}。{ベネフィット1}、{ベネフィット2}。{根拠機能}で手間なく続きます。{信頼要素}。」

### 主要キーワード（primary keywords）
- 方針：
  - “機能名”より“検索される言葉”を優先
  - 3階層で用意：主要（2〜3）/準主要（5〜8）/補助（10〜20）
  - タイトル/見出し/H1付近に主要キーワードを自然に含める
  - キーワードの詰め込みは避ける

主要（2〜3）：
- 筋トレ 記録
- ワークアウト 記録
- トレーニング ログ

準主要（5〜8）：
- 筋トレ アプリ
- ワークアウト アプリ
- トレーニング 記録 アプリ
- カレンダー 記録
- 筋トレ 履歴
- 振り返り アプリ
- 種目 お気に入り
- iCloud 保存（※表現は実態に合わせる）

補助（10〜20）：
- ジム 記録
- 自重 トレーニング 記録
- ルーティン 記録
- 今日のトレーニング
- 種目 検索
- セット 記録
- 重量 回数 記録
- シンプル アプリ
- 続けやすい
- iPhone 筋トレ 記録

### OGP方針（Open Graph / SNS共有）
- 目的：SNS/チャット共有での見栄え・誤解防止
- ルール：
  - og:title：titleを短く再編集（Biggrを含める）
  - og:description：meta descriptionを短く（60〜90文字目安）
  - og:image：実画面が伝わる1枚（推奨 1200×630）
    - 1画面 + 余白。UIが小さくなりすぎない構図
    - 文字はキャッチ1行まで（入れるなら）
  - og:type：website
  - twitter:card：summary_large_image
  - 画像はローカライズを検討（最低でも日本語版1枚）

## Accessibility / Performance（LP）

### 目標 LCP / CLS（Core Web Vitals）
- 指標は「実ユーザー（field data）の75パーセンタイル」を基準にする（モバイル/デスクトップ別）
- LCP（Largest Contentful Paint）：<= 2.5 秒
- CLS（Cumulative Layout Shift）：<= 0.1
- 補足：Lighthouse等のラボ計測は参考値。最終評価はフィールドデータを優先する。

### コントラスト基準（WCAG AA）
- 通常テキスト：4.5:1 以上
- 大きいテキスト：3:1 以上
- UIコンポーネント/アイコン等の非テキスト要素：3:1 以上

### フォントサイズ最小値
- 本文（body）：16px 以上
- 入力欄（input/textarea）：16px 以上（iOSの自動ズーム回避も兼ねる）
- 補助情報（注記・メタ情報など）：原則 14px 以上（長文には使わない）

## Localization / 用語統一
- 対応言語：日本語 / 英語
- 用語の一貫性：
  - 同じ概念は同じ言葉で統一する（例：記録 / 履歴 / 振り返り）
  - CTA文言は全ページで統一する（例：「App Storeで入手」）

---

## Tech Stack / Build
- Tech Stack：静的HTML（HTML / CSS / JavaScript）
- 方針：
  - 依存を増やさず、読み込みを軽く保つ
  - 画像最適化（サイズ/形式）を優先して LCP を改善する
- ビルド手順（要約）：静的ファイルをそのまま配信（ビルド不要）
- デプロイ手順（要約）：ホスティング（例：GitHub Pages等）に静的ファイルを配置して公開

## Bilingual LP（日本語 / 英語）方針

### 目的
- 日本語/英語どちらのユーザーにも、同じ体験・同じ情報設計で提供する。
- 翻訳ではなく「各言語で自然な短い文」を優先する（直訳しない）。

---

### ページ構成（推奨）
- `/ja/`：日本語LP（デフォルト）
- `/en/`：英語LP
- `/`：`/ja/`へリダイレクト、または言語選択（要件により選ぶ）
- Footer に言語切替（JP / EN）を常設する

---

### hreflang / canonical（SEO）
- 各ページに hreflang を必ず付与する
  - 日本語ページ：`hreflang="ja"`
  - 英語ページ：`hreflang="en"`
  - 言語未指定の共通：`hreflang="x-default"`（必要な場合のみ）
- canonical は「同一言語の自分自身」を指す（言語間でcanonicalを跨がせない）
  - 例：`/ja/` の canonical は `/ja/`
  - 例：`/en/` の canonical は `/en/`

---

### タイトル / description（言語ごとに最適化）
- title/description は各言語で自然な表現にする（直訳しない）
- 日本語は「筋トレ記録/ワークアウト記録」の主要語を自然に含める
- 英語は “workout tracker / workout log / gym log” など一般的表現を使う
- 各言語で CTA 文言も固定して統一する（下記参照）

---

### OGP（言語別）
- og:title / og:description はページ言語に合わせる
- og:image は基本は共通でもよいが、可能なら言語別を用意する
  - 例：`og:image`（ja）と（en）でキャッチの言語を合わせる
- twitter:card は `summary_large_image` を使用する

---

### Language Switch（言語切替）
- ヘッダー/フッターに「JP / EN」を明確に表示
- 現在の言語を強調し、他言語へワンクリックで移動できること
- 切替は同じセクションへ遷移する必要はない（まずはトップ遷移でOK）

---

### 文章ルール（翻訳運用）
- 日本語→英語の機械的な直訳を禁止（短く自然な文に再構成）
- 用語は各言語で統一する（同じ概念を言い換えない）
- 長文より短文。1文は短く、形容詞を増やさない

---

### 用語・CTAの固定（推奨）
- App Name：Biggr（表記固定）
- 日本語 Primary CTA：`App Storeで入手`
- 英語 Primary CTA：`Download on the App Store`
- Secondary CTA（任意）：
  - 日本語：`特徴を見る`
  - 英語：`See features`

---

### HTML実装（最低限）
- `<html lang="ja">` / `<html lang="en">` を正しく設定する
- 主要見出し構造（H1→H2→H3）を両言語で揃える
- 言語別に meta title/description/OGP を持つ（共通流用しない）

---

## Non-goals
- このLPでやらないこと：
  - ブログ / 記事の継続運用を前提にしない
  - 過度なアニメーションや重い演出で体験を損ねない
  - 証明できない比較最上級（「最も」「唯一」等）を前提にしない
